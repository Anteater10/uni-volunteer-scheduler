# v1.3 Integration onto Milestone1.2Final (branch: `v1.3-check`)

**Date:** 2026-04-17
**Source branch:** `v1.3` (older commit base, forked from `c27cd25`)
**Target branch:** `v1.3-check` (new, created from `Milestone1.2Final` HEAD `6933cb0`)
**Common ancestor:** `c27cd25` — "merge: combine v1.2-participant with newOrganiser"

## Goal

Bring v1.3's feature set onto the verified Milestone1.2Final base **without losing**:

- The dual-mode Mailpit / SES / SendGrid email wiring added on Milestone1.2Final
  (commit `326e19f`), specifically `_send_email_via_sendgrid` in `celery_app.py`
  and all callers of it in `services/invite.py` and `routers/admin.py`.
- The cross-role Playwright regression coverage (`e2e/cross-role.spec.js`,
  ~5 scenarios × 6 browser projects).
- The admin event slot management UI additions on `EventsSection.jsx`,
  `AdminEventPage.jsx`, `AdminLayout.jsx`, plus the organizer dashboard
  enhancements on `OrganizerDashboard.jsx` / `EventDetailPage.jsx`.
- The v1.2-prod docs close-out (README, ROADMAP, STATE, smoke checklist,
  INTEG-05 bug log).

## Non-goals

- No merging back to `main` yet. `v1.3-check` stays local/origin-only until the
  user verifies each cherry-picked phase and says to ship.
- No ripping out or refactoring any Milestone1.2Final-era code "on the way in."
  v1.3 features land as cleanly as possible; unrelated improvements are out of
  scope.

## Scope — what comes in vs. what is deferred

v1.3 delivered 9 feature phases (21 through 29). This integration brings in 7:

| Phase | Name                                      | Status       |
| ----- | ----------------------------------------- | ------------ |
| 21    | Orientation credits engine                | **Include**  |
| 22    | Custom form fields                        | **Include**  |
| 23    | Recurring event duplication               | **Include**  |
| 24    | Scheduled reminder emails                 | **Include**  |
| 25    | Waitlist + auto-promote                   | **Include**  |
| 26    | Broadcast messages                        | **Include**  |
| 27    | SMS reminders via AWS SNS                 | **Skip**     |
| 28    | QR check-in (v1.3 model)                  | **Skip**     |
| 29    | Signup window + swap + hide-past toggle   | **Include**  |

**Phase 27 (SMS)** is skipped because AWS SNS is not provisioned and we have no
way to verify SMS end-to-end in dev. Migration `0016_volunteer_preferences`
(authored by Phase 24) includes dormant `sms_opt_in` and `phone_e164` columns
that ride along — that is fine, they stay unused until 27 lands later.

**Phase 28 (v1.3 QR model)** is skipped because v1.3 built the wrong shape for
this project:

- v1.3: every volunteer gets a personal QR in their confirmation email, and
  the organizer uses a camera-based `QRScanner` component to scan them.
- What we want (tracked as a separate follow-up phase): **one QR per event**,
  displayed on the organizer's screen behind a "Show check-in QR" button.
  Volunteers scan it with their phone and land on a `/check-in/{event_id}`
  page where they self-identify. That rebuild is queued as a post-integration
  task and will happen on this same branch after all 7 cherry-picks are in.

## Strategy — phase-by-phase cherry-pick

1. Branch `v1.3-check` already created from `Milestone1.2Final` HEAD.
2. For each included phase N in order `[21, 22, 23, 24, 25, 26, 29]`:
   1. List commits from v1.3 whose subject matches `(N)` or that are required
      seed docs for v1.3 (`docs(v1.3)` — the roadmap/requirements seed commit
      `e123d54`, included with Phase 21).
   2. Cherry-pick them in chronological order with `git cherry-pick -x`
      (so the target branch records the v1.3 commit SHA for traceability).
   3. Resolve conflicts — see "Conflict handling" below.
   4. Run the verification bundle (see "Verification bar").
   5. Push `v1.3-check` to `origin` so the user can inspect the phase.
3. Phases 27 and 28 commits are **skipped entirely** — not cherry-picked and
   not referenced in commit messages.
4. After Phase 29 completes verification, run the "final integration bundle"
   and hand off for manual smoke before the organizer-QR follow-up phase.

## Conflict handling posture

- **Line-level / rename / straightforward overlap:** resolve in-place during
  the cherry-pick. If a v1.3 file edit targets a file we've heavily modified
  on Milestone1.2Final (e.g., `EventsSection.jsx`, `EventDetailPage.jsx`,
  `api.js`, `App.jsx`, `AdminLayout.jsx`), merge both sides of the change so
  v1.3's new feature (e.g., DuplicateEventDrawer button) co-exists with our
  slot management additions.
- **Best-effort + ask:** attempt to resolve conflicts and get verification
  green. If stuck or if the fix would significantly alter v1.3's intent,
  pause and summarize options for the user.
- **Hard stop — design reshape:** if resolving a conflict would require
  **redesigning** a v1.3 feature (not just reconciling line edits), stop and
  ask. Examples: if v1.3's reminder_service bypasses `_send_email_via_sendgrid`
  entirely and we'd have to rewrite its send path from scratch, that is a
  design decision and pauses the run.
- **Migration chain:** v1.3's migrations `0014 → 0017` stack cleanly on our
  `0013`. If any of them references a down_revision from a phase we're
  skipping (27), stitch them in-place (no new revision IDs; only retarget
  `down_revision` if the skipped migration sat in the middle).

## Verification bar per phase ("B")

After each phase's commits land and conflicts are resolved:

1. `alembic upgrade head` must run clean inside the backend container.
2. Backend must boot under `docker compose up backend` without exceptions
   (health endpoint returns 200).
3. Frontend must pass `npm run build` (catches import / syntax breakage
   without needing a live dev server).
4. `pytest -q` must be green (run inside a one-off container on the
   `uni-volunteer-scheduler_default` network per `CLAUDE.md`).
5. `npm run test -- --run` must be green.

Only when all five are green does the phase commit move get pushed and the
next phase start. Failure → see "Conflict handling posture" above.

## Final integration bundle

After all seven phases are in:

1. Destroy and recreate the dev DB to verify a **clean** `alembic upgrade head`
   from empty — catches out-of-order migration issues.
2. Full backend `pytest -q` + frontend `npm run test -- --run`.
3. Full stack up (`docker compose up`).
4. Playwright: existing cross-role spec must still pass. Any v1.3-added
   Playwright scaffold (`9be8eb6 test(29): v1.3 cross-feature playwright
   scaffold`) must either pass or have SMS/QR scenarios excluded (since we
   skipped 27 and 28).
5. Hand the branch + a written summary to the user for the one-time manual
   smoke.

## Branch + push cadence

- `v1.3-check` lives on origin after the spec commit.
- Push after each green phase. The user sees each phase as its own force-of-
  habit `git pull` checkpoint.
- No PR opened. The user opens any PR themselves per their standing
  "push branches, don't open PRs" rule.

## Per-project-rule exceptions on this branch only

- The project's standing rule is "never commit `.planning/` or other AI
  scaffolding." For `v1.3-check` **only**, the user has explicitly authorized
  committing v1.3's `.planning/` files (CONTEXT/PLAN/SUMMARY per phase,
  REQUIREMENTS-v1.3.md, ROADMAP.md, config.json) so the full v1.3 history is
  preserved for verification. Cherry-picks do **not** filter `.planning/`
  paths. This exception does not apply to any other branch.

## Post-integration follow-up (tracked separately)

After all seven phases are in and verified, rebuild check-in with the correct
product shape:

- Organizer-side button on an event → modal/drawer displaying one QR for that
  event.
- QR encodes `/check-in/{event_id}` (or a signed short-lived token variant).
- Volunteers scan → land on a lightweight self-identification page →
  `check_in_service` records attendance.
- Do not embed QR in confirmation emails.
- Do not ship the v1.3 `QRScanner` camera component.

This rebuild happens on `v1.3-check` after integration verification is green.
