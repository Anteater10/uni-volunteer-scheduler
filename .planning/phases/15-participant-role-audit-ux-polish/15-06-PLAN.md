## Trust Boundaries

|----------|-------------|

| signupQ error payload → page render | Error kind string from server (OUTSIDE_WINDOW / WRONG_VENUE_CODE / INVALID_TRANSITION) |
| Orientation modal user assertion → signup submit | Claim propagated to server; server holds authoritative record |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-06-01 | Input Validation | Venue code input | mitigate | `maxLength={4}` + `pattern="[0-9]*"` client-side hints; `inputMode="numeric"` for mobile keyboard. Server-side validation is authoritative (api.js unchanged per D-14). |
| T-15-06-02 | Information Disclosure | Error copy surfaces state to user | mitigate | UI-SPEC copy is deterministic literals. Removes prior bare `<p>Could not load signup details.</p>` which could surface raw error.message. |

Backend concern (CSP frame-ancestors / X-Frame-Options). If PART-AUDIT surfaces that header is missing, file a backlog entry — do NOT modify backend here (D-14). |
| T-15-06-04 | Tampering | Orientation modal user assertion | accept | User self-reports orientation status by clicking primary button. Server enforces reality via attendance records; modal is a soft gate per CONTEXT D-14 / PART-06. Unchanged behavior. |
| T-15-06-05 | Spoofing | FieldError aria-describedby wiring | mitigate | `id="venue-code-error"` + `aria-describedby="venue-code-error"` links screen-reader announcement to input. Standard ARIA pattern; cannot be exploited. |
