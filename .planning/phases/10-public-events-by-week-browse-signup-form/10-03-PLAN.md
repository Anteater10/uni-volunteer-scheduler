**State machine (per D-11):**
```javascript
const [step, setStep] = useState("browse");
// "browse" | "form" | "checking-orientation" | "orientation-warning" | "submitting" | "success"
const [selectedSlotIds, setSelectedSlotIds] = useState(new Set());
const [identity, setIdentity] = useState({ first_name: "", last_name: "", email: "", phone: "" });
const [formErrors, setFormErrors] = useState({});
const [submitError, setSubmitError] = useState(null);
const [successData, setSuccessData] = useState(null);
const [highlightOrientation, setHighlightOrientation] = useState(false);
```

**Page sections:**

1. **Event header:** PageHeader with event title, school, date range (formatted). Back link to /events.

2. **Slot list per D-03:**
   - Group slots by slot_type: "Orientation Slots" section, then "Period Slots" section
   - Each slot is a Card with a checkbox (input type="checkbox", h-5 w-5 min-w-5)
   - Slot card shows: date (formatted), time range, location, capacity ("X/Y spots left")
   - Full slots (filled >= capacity): checkbox disabled, "Full" badge, muted styling
   - If highlightOrientation is true, orientation slot cards get a ring highlight (ring-2 ring-[var(--color-primary)])
   - Toggle function: `toggleSlot(id)` adds/removes from selectedSlotIds Set
   - Touch targets >= 44px per D-06: the entire card row is clickable (not just the tiny checkbox)

3. **Identity form (shows when selectedSlotIds.size > 0 AND step is "browse" or "form"):**
   - On first slot selection, auto-transition step from "browse" to "form"
   - Four fields: first_name (Input, required), last_name (Input, required), email (Input type="email", required), phone (Input type="tel", placeholder="(555) 123-4567" per D-09)
   - Label + FieldError for each field
   - Client-side validation on submit: all fields required, email must match basic regex (/^[^\s@]+@[^\s@]+\.[^\s@]+$/), phone must have 10+ digits (strip non-digits, check length >= 10)
   - "Sign up" Button (primary, full-width, min-h-11)

4. **Submit handler (handleSubmit):**
   - Validate client-side first. If errors, setFormErrors and return.
   - Build a slotMap from the event data for quick lookup: `{id: slot}`
   - Check if any selected slot is slot_type="period" AND no selected slot is slot_type="orientation" per D-04
   - If so: setStep("checking-orientation"), call api.public.orientationStatus(identity.email)
     - If has_attended_orientation === true: skip modal, proceed to submitSignup()
     - If has_attended_orientation === false: setStep("orientation-warning"), return (modal will show)
     - On API error: log nothing (no console.log of email per threat model), proceed to submitSignup() -- don't block signup over a failed check
   - If no orientation check needed: proceed directly to submitSignup()

5. **submitSignup() function:**
   - setStep("submitting")
   - Call api.public.createSignup({ ...identity, slot_ids: [...selectedSlotIds] })
   - On success: setSuccessData({ ...response, slots: selectedSlots }), setStep("success")
   - On error per D-08:
     - err.status === 429: toast.error("Too many submissions. Please wait a moment and try again.")
     - err.status === 422: parse field errors from err.response.data.detail and set formErrors. setStep("form")
     - err.message containing "capacity" or "full": toast.error("One or more selected slots are now full. Please pick different slots."). setStep("browse"). Refetch event data (invalidate query).
     - Other: toast.error(err.message). setStep("form")

6. **Loading/error states:**
   - useQuery for api.public.getEvent(eventId) with queryKey ["publicEvent", eventId]
   - isPending: show Skeleton cards
   - error: EmptyState with error message + Retry button

IMPORTANT per threat model: Do NOT console.log, localStorage.setItem, or sessionStorage.setItem any PII (email, phone, name). The identity state lives only in React component state and is cleared on unmount or form reset.
