## Why

Current app covers the baseline volunteer scheduling loop. SciTrek still relies on an external scheduling tool for several capabilities that would be strictly better in-app because we control the domain rules — especially the cross-module orientation credit rule (see `.planning/notes/2026-04-15-scitrek-orientation-rule.md`) that no off-the-shelf tool models. Hung wants full parity as the v1.3 direction.

AWS context: ~$94k in promo credits available. SNS (SMS), SES (email), RDS/ECS/Lambda all fit comfortably — credit burn is not a constraint for this milestone.

## Scope — 9 capability gaps (ordered by proposed phase sequence)

1. **Orientation credit engine** — redesign orientation-warning to check historical attendance across all weeks for the same module family (wk4 attendance satisfies wk6 of the same module). Highest value, touches the domain model. Load-bearing SciTrek rule.
2. **Recurring event duplication** — admin action "duplicate this event to weeks N…M." Depends on Templates CRUD (Phase 17) shipping.
3. **Scheduled reminder emails** — Celery Beat job, 24h + 2h pre-event. Infra already running.
4. **Waitlist + auto-promote** — new signup state (`waitlisted`), promotion hook fires on cancel, email notification when promoted.
5. **Broadcast messages** — organizer/admin → email all signups for an event ("parking moved to Lot 22"). Reuses email infra.
6. **SMS reminders + no-show nudges** — AWS SNS integration, opt-in phone field on signup form. ~$0.0075/SMS.
7. **QR check-in** — generate per-signup QR on confirmation email; organizer scans at venue. Faster than magic-link-on-phone when a line forms.
8. **Slot swap / trade** — atomic move between slots; keeps audit trail, preserves orientation credit.
9. **Add-to-calendar (ICS)** — smallest. Could fold into Phase 15 PART-13 instead of waiting for v1.3.

## Explicitly out of scope

- Payments / donations / tickets / auctions — SciTrek doesn't take money.
- Billing tiers, admin quotas, email/SMS quotas — single-tenant doesn't meter itself.
- SSO, custom domains, branding, portal pages for external orgs — single-tenant.
- Multi-tenant / SaaS anything — Andy's "UCSB-wide rollout" idea is far-future, not v1.3.
